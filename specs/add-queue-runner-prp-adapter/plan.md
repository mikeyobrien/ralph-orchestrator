# Queue Runner + PRP Adapter Implementation Plan

> REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task.

**Goal:** Add a generic Ralph queue runner with a built-in PRP adapter so queued PRPs run in deterministic worktrees, merge automatically, archive on success, recover cleanly after interruption, and advance sequentially without an external shell wrapper.

**Scope:** Included: queue state and CLI, deterministic PRP worktrees and branches, explicit merge metadata for non-`ralph/*` branches, nested child-worktree consolidation before final merge, PRP finalization and recovery, and automated tests and docs in this repo. Excluded: web dashboard queue controls, non-PRP adapters beyond the generic contract, and downstream consumer repo prompt/template changes.

**Architecture / Approach:** Keep the backlog controller in `ralph-cli`, not in the core event loop. Reuse existing `ralph-core` primitives for worktree management, loop registry, merge queue persistence, and completion handling, but extend them so queue-managed loops can carry explicit source and target branch metadata instead of relying on the current `ralph/{loop_id}` convention. The queue runner should acquire the repo lock once, run one explicit worktree loop at a time via `run_loop_impl`, process merges itself after each successful worktree completion, finalize the PRP on the target branch, then continue to the next queued item.

**Tech Stack:** Rust, `clap`, `serde`, `serde_json`, existing `ralph-core` and `ralph-cli` unit and integration tests, git worktree and merge commands, built-in `merge-loop` preset, queue and merge JSON files under `.ralph/`.

**Dependencies / Constraints:**
- Preserve existing `ralph run`, `ralph loops`, and merge-queue behavior for legacy non-queue loops.
- Reuse repo-local `.worktrees/` for implementation work in this repository. `git check-ignore -v .worktrees/` already passes in this repo.
- Queue progression is strictly sequential: do not start the next PRP until the current one is merged, finalized, and cleaned up.
- `needs_review` is a hard stop, not an auto-skip.
- PRP top-level identity must stay deterministic: worktree root `.ralph-worktrees/{PRP_NAME}` and branch `prp/{PRP_NAME without leading PRP-}`.
- Preserve the current parent-child behavior by consolidating nested child worktrees back into the parent PRP branch before the parent merges to `main`.
- Do not route queue execution through `run_command`; that path owns lock acquisition and autogenerated worktree naming for interactive CLI runs and will fight the deterministic queue flow.

**Definition of Done:**
- `ralph queue run --adapter prp --source <dir>` can process PRPs sequentially from a repo-owned queue state file.
- Queue-managed PRP runs use deterministic worktree and branch names and merge successfully through Ralph's merge flow.
- A merged PRP is finalized by archiving its markdown from `remaining_work` to `completed`, then the queue advances automatically.
- Interrupted queue runs can resume the same PRP without redoing completed finalization work.
- Existing non-queue merge flows still pass.
- Automated tests cover queue state, merge metadata propagation, PRP recovery, nested child consolidation, and next-item progression.

## Locked Design Decisions

These choices are fixed for this implementation and should not be re-opened during coding.

### Queue State

- Persist queue state at `.ralph/queue-state.json`.
- Use a snapshot file, not JSONL.
- Top-level shape:

```json
{
  "version": 1,
  "adapter": "prp",
  "source": "PRPs/remaining_work",
  "items": [
    {
      "item_id": "PRP-003-adaptive-worker-scaling",
      "state": "pending",
      "source_path": "PRPs/remaining_work/PRP-003-adaptive-worker-scaling.md",
      "completed_path": "PRPs/completed/PRP-003-adaptive-worker-scaling.md",
      "loop_id": "prp-PRP-003-adaptive-worker-scaling",
      "queue_item_id": "PRP-003-adaptive-worker-scaling",
      "source_branch": "prp/003-adaptive-worker-scaling",
      "target_branch": "main",
      "worktree_path": ".ralph-worktrees/PRP-003-adaptive-worker-scaling",
      "parent_loop_id": null,
      "merge_commit": null,
      "last_error": null
    }
  ]
}
```

- Item states are exactly:
  - `pending`
  - `running`
  - `merging`
  - `finalizing`
  - `completed`
  - `needs_review`
  - `failed`

### Queue CLI

- Add a new top-level command namespace: `ralph queue`.
- First supported commands:
  - `ralph queue run --adapter prp --source <dir>`
  - `ralph queue run --continue`
  - `ralph queue status`
  - `ralph queue status --json`
- `queue run --continue` resumes from `.ralph/queue-state.json`; it does not rely on the legacy primary-loop scratchpad check in `run_command()`.

### Adapter Contract

- Keep adapters in `ralph-cli`, not `ralph-core`.
- Do not build a plugin loader. The queue runner can dispatch built-in adapters with an enum or simple match.
- The first adapter is `prp`.
- Adapter responsibilities are:
  - discover items
  - order items
  - compute deterministic identity and paths
  - provide prompt-file and env values for loop execution
  - determine whether an item is already finalized
  - finalize a merged item

### PRP Naming

- For item stem `PRP-003-adaptive-worker-scaling`:
  - `item_id = PRP-003-adaptive-worker-scaling`
  - `loop_id = prp-PRP-003-adaptive-worker-scaling`
  - `source_branch = prp/003-adaptive-worker-scaling`
  - `target_branch = main`
  - `worktree_path = <repo>/.ralph-worktrees/PRP-003-adaptive-worker-scaling`
- Branch-name derivation rule:
  - If the filename stem starts with `PRP-`, remove that exact prefix and lowercase nothing.
  - Otherwise use the full stem as-is after `prp/`.

### Environment Contract

- Queue-managed worktree loops must receive:
  - `RALPH_QUEUE_MANAGED=1`
  - `RALPH_QUEUE_ADAPTER=<adapter>`
  - `RALPH_QUEUE_ITEM_ID=<item_id>`
  - `RALPH_QUEUE_LOOP_ID=<loop_id>`
  - `RALPH_QUEUE_SOURCE_PATH=<absolute source markdown path>`
  - `RALPH_QUEUE_COMPLETED_PATH=<absolute completed markdown path>`
  - `RALPH_QUEUE_SOURCE_BRANCH=<branch being merged>`
  - `RALPH_QUEUE_TARGET_BRANCH=<branch receiving merge>`
  - `RALPH_QUEUE_WORKTREE=<absolute worktree path>`
  - `RALPH_QUEUE_PARENT_LOOP_ID=<optional>`

- Merge-loop subprocesses must receive:
  - `RALPH_MERGE_LOOP_ID=<loop_id>`
  - `RALPH_MERGE_SOURCE_BRANCH=<source_branch>`
  - `RALPH_MERGE_TARGET_BRANCH=<target_branch>`
  - `RALPH_MERGE_WORKTREE_PATH=<absolute worktree path>`
  - `RALPH_MERGE_QUEUE_ITEM_ID=<item_id>`

### Merge Behavior

- Merge metadata is explicit. Queue-managed merges must not reconstruct branch names from `loop_id`.
- Legacy behavior stays as fallback:
  - source branch defaults to `ralph/{loop_id}`
  - target branch defaults to `main`
  - worktree defaults to `.worktrees/{loop_id}`

### Queue Runner Control Flow

- The queue runner must acquire the repo `LoopLock` once at the start of `queue run`.
- It should then run each queued item by calling `crate::loop_runner::run_loop_impl()` directly with a `LoopContext::worktree(...)`.
- For each item, the queue runner must register a `LoopEntry` before calling `run_loop_impl()`.
- `run_loop_impl()` will deregister the current process on termination, so the queue runner must re-register for the next item.
- After a queue-managed worktree loop exits with `CompletionPromise`, the queue runner must call `process_pending_merges_cli(repo_root)` itself, because `run_loop_impl()` only auto-processes merges on primary-loop completion.

### Nested Child Worktrees

- Preserve the current wrapper behavior:
  - Before a parent PRP merge into `main`, merge nested child worktree branches back into the parent PRP branch.
  - If any child merge conflicts, abort that child merge, leave all relevant worktrees intact, and mark the parent queue item `needs_review`.

## Execution Flow

Implement exactly this runtime sequence for `ralph queue run --adapter prp --source <dir>`:

1. Load config via the same preflight loader used by `run_command()`.
2. Acquire `LoopLock` on the repo root for the whole queue run.
3. Load or create `.ralph/queue-state.json`.
4. Discover PRP items from the adapter and reconcile queue state with filesystem truth.
5. Pick the first non-terminal item:
   - `pending`
   - `running`
   - `merging`
   - `finalizing`
6. If the item is already merged and finalized according to adapter recovery logic, mark it `completed` and move on.
7. If the item is `pending` or `running`, ensure deterministic worktree and branch exist.
8. Register a `LoopEntry` using the item's fixed `loop_id`, `worktree_path`, `queue_item_id`, and optional `parent_loop_id`.
9. Execute the item with `run_loop_impl()` in headless mode using `LoopContext::worktree`.
10. If the loop exits unsuccessfully, mark the queue item `failed` or `needs_review` and stop.
11. If the loop exits with `CompletionPromise`, mark queue state `merging`, then call `process_pending_merges_cli(repo_root)`.
12. Poll merge queue state for this `loop_id` until:
    - `merged`: continue to adapter finalization
    - `needs_review`: mark queue item `needs_review` and stop
13. On `merged`, run adapter finalization on the target branch.
14. If finalization succeeds, mark queue item `completed` and continue to the next item.
15. If finalization fails, mark queue item `finalizing` or `failed` with `last_error` and stop.

**Steps:**
1. Implement `queue::execute(...)` with `run` and `status` subcommands.
2. For `queue run`:
   - load config through the existing preflight loader
   - acquire `LoopLock` once for the repo root
   - load or create queue state
3. For each selected queue item:
   - ensure worktree root is ignored
   - create or attach deterministic worktree and branch
   - construct `LoopContext::worktree(loop_id, worktree_path, repo_root)`
   - write queue state `running`
   - register a `LoopEntry` with the fixed queue loop metadata
4. Execute the item by calling `crate::loop_runner::run_loop_impl(...)` directly with:
   - `resume = false` for first run
   - `resume = true` and `resume_loop_id = Some(loop_id.clone())` for `queue run --continue` on an already-running item
   - `enable_tui = false`
   - `enable_rpc = false`
   - `auto_merge_override = Some(true)`
5. After `CompletionPromise`, mark queue state `merging`, then call `crate::loop_runner::process_pending_merges_cli(repo_root)`.
6. Poll merge queue state until terminal:
   - `merged` -> queue state `finalizing`
   - `needs_review` -> queue state `needs_review`, stop

- Do not call `run_command()`. It will try to own lock acquisition, prompt normalization, and autogenerated worktree behavior that conflicts with queue-managed deterministic execution.
- Because `run_loop_impl()` deregisters the current process on termination, the queue runner must re-register a fresh `LoopEntry` for each subsequent item.
- `queue status` should read only `.ralph/queue-state.json`; do not derive status by re-scanning the filesystem every time.

## Risks and Assumptions
- The highest-risk area is merge behavior drift: queue-managed metadata must not break existing legacy merge entries or `ralph loops` commands.
- The plan assumes the queue runner can safely hold the repo `LoopLock` for the whole sequential run while still executing worktree loops in-process via `run_loop_impl()`.
- The plan assumes `.ralph-worktrees/` remains a consumer-repo choice owned by the PRP adapter, while this repo's contributor worktree for implementation stays under `.worktrees/`.
- The plan assumes child-worktree consolidation is sufficient for v1 and that full nested queue inheritance can be deferred.
- The plan assumes queue execution can stay in `ralph-cli` without adding new web or RPC control-plane endpoints.

## Deferred / Out of Scope
- Updating `reevr` or other consuming repos to remove their current PRP shell wrappers.
- Web dashboard support for queue state and queue-item controls.
- Additional queue adapters beyond PRP.
- Parallel queue-item execution.
- Plugin-based adapter loading.
